#房间解谜游戏·策划案
## 任务ID：JJC-20260531-011
**产出日期：** 2026-05-31
**基于研究：** Hollow Halls 调研 + 类银河恶魔城无战斗解谜游戏通用设计知识

---

## 一、游戏概述

- **游戏名称：** 《暗室》（暂定）
- **类型：** 2D类银河恶魔城 · 无战斗房间解谜
- **核心特色：** 玩家通过切换房间内的"预制件槽位"（Switchable Tiles）来重塑房间结构，从而开辟通往出口的道路。无敌人、无战斗、无升级成长，核心体验是"空间逻辑推理 + 房间拓扑变换"。
- **目标用户：** 解谜游戏爱好者、独立游戏玩家、类银河恶魔城轻度玩家
- **原型参考：** Hollow Halls（无战斗探索解谜）、Dead Cells（房间预制拼贴）、Gorogoa（空间切换解谜）

> **注：** 关于 Hollow Halls 的调研结果——它是一款**恐怖氛围**的短篇独立游戏，聚焦**故事与氛围**，非战斗导向。玩家被困在一座废弃设施中，通过探索和解谜来拼凑故事并逃出设施。**不允许多存档**。这与"房间预制件切换"机制并不完全吻合，说明皇上的原型描述可能是对 Hollow Halls 机制的某种想象性重构，或参考了其他类似机制的游戏（如 Dead Cells 的房间生成方式）。策划案以"房间预制件切换"为核心机制进行设计。

## 二、核心玩法

### 2.1 主要机制

| 要素 | 说明 |
|------|------|
| **预制件槽位（SwitchSlot）** | 房间中特定位置设有槽位，每个槽位包含2~4 个预制件选项 |
| **预制件类型** | 地板（可通过）/实墙（不可通过）/机关门（满足条件开启）/透明墙（可穿行） |
| **玩家操作** | 走近槽位 → 屏幕提示当前选项 → 按键循环切换预制件 → 房间结构实时变化 |
| **通关条件** | 切换出正确配置，使玩家可以从入口到达出口 |
| **失败条件** | **无**——可无限切换，可随时重置（Reset）房间 |
| **重置机制** | 按R键将所有槽位恢复到房间初始状态 |

### 2.2 核心循环

```
[进入房间] → [观察环境/槽位位置] → [尝试切换1个或多个槽位] → [验证连通性（视觉判断）] → [到达出口] → [进入下一房间]
```

-核心循环强调**即时反馈**：每次切换后，玩家立刻能看出道路是否打通
- 验证连通性是纯视觉的（无需寻路算法提示），通过地板颜色/发光指引

### 2.3 地图/关卡结构

- **整体结构：** 类银河恶魔城的横向展开地图，非线性的区域连接
- **章节设计：** 共3个章节，每章5~8个房间
  - **第一章·觉醒：** 引入基础切换（地板/墙），教学导向，5个房间
  - **第二章·深掘：**引入机关门、时序开关（某槽位切换后另一槽位状态改变），6个房间
  - **第三章·迷途：** 引入联动开关（需要同时满足多个槽位配置）、伪装地板（视觉欺骗），8个房间
- **门控设计：** 完成上一章节全部房间后解锁下一章节
- **能力解锁：** 无（保持无成长驱动，所有机制在第一章逐步引入）

## 三、特色设计

### 3.1 无战斗类银河恶魔城的设计哲学

**为什么无战斗？**
- 降低开发复杂度（无AI、无战斗平衡、无数值设计）
- 将认知压力集中在空间逻辑上，而非操作压力
- 适合解谜玩家核心诉求：安静思考

**与传统类银河恶魔城的区别：**

| 维度 | 传统类银河恶魔城 | 本游戏 |
|------|----------------|--------|
| 驱动类型 | 能力成长驱动（获得新能力→探索新区域） | **空间逻辑驱动**（切换房间拓扑→打通道路） |
| 进度机制 | 能力解锁门控 | 房间完成度解锁 |
| 玩家成长 | 数值/技能成长 | 无成长，纯智力挑战 |
| 探索感 | 地图逐渐展开的惊喜感 | 房间内部的拓扑重构惊喜感 |

**在没有"成长驱动"的情况下保持吸引力：**
1. **机制层层叠加：** 每章引入新机制，后一个房间需要同时运用已有+新机制
2. **拓扑美学：** 房间切换带来的空间重构感本身就是奖励（类似Gorogoa的"画卷展开"体验）
3. **无压力探索：** 无死亡惩罚，允许无限重置，降低进入门槛
4. **叙事碎片：** 通过环境物品（文件、照片、涂鸦）暗示故事背景，满足叙事需求

### 3.2 房间预制件系统详解

**槽位类型（SwitchSlot）：**

| 类型 | 行为 |
|------|------|
| **ToggleSlot** | 二选一切换（A↔ B） |
| **CycleSlot** | 循环切换（A → B → C → A） |
| **ConditionalSlot** | 依赖另一槽位状态，解锁更多选项 |
| **LockedSlot** | 需要先完成特定目标才可激活 |

**预制件类型（PrefabOption）：**

| 类型 | 视觉效果 | 行为 |
|------|--------|------|
| **SolidWall** | 深灰色实墙 | 阻挡移动 |
| **Floor** | 地板色，可通行 | 正常地面 |
| **GlassWall** | 半透明蓝色 | 可通行但视觉上有阻挡感 |
| **Door** | 门框（开/闭状态） | 关闭时不可通行，开启后可通行 |
| **CrumblingFloor** | 碎裂地板纹理 | 踩上去后碎裂消失（一次性） |
| **FakeFloor** | 与普通地板外观相同 | 实际为实墙，玩家会踩空/视觉反馈 |
| **PressurePlate** | 踩下时触发 | 联动其他槽位变化 |

**切换动画和反馈：**
- 切换动画：预制件淡出淡入（0.2s），伴随切换音效
- 视觉反馈：槽位位置有微弱发光标记（引导玩家注意力）
- 连通性反馈：当出口路径连通时，出口发出脉冲光提示

### 3.3 难度曲线设计

**前3个房间的教学引导：**

| 房间 | 教学目标 | 槽位配置 |
|------|--------|--------|
| 1-1 | 理解槽位概念 | 1个ToggleSlot，2选项（墙/地板），切换后即可通行 |
| 1-2 | 理解有多个槽位 | 2个ToggleSlot，需同时切换正确组合 |
| 1-3 | 理解出口位置需要特定配置 | 2个ToggleSlot + 1个CycleSlot，出口被实墙堵住 |

**新机制引入节奏：**

| 章节 | 引入机制 | 示例 |
|------|--------|------|
| 第一章 | 基础切换（地板/墙）、多槽位组合 | 5个房间 |
| 第二章 | 机关门（需要先切换另一槽位）、ConditionalSlot | 6个房间 |
| 第三章 | 联动开关（多槽位联动）、伪装地板（Crumbling/Fake） | 8个房间 |

**房间复杂度升级路径：**
- 槽位数量：1 → 2 → 3 → 4（逐章增加）
- 选项数量：2 → 3 → 4（选项越多推理越复杂）
- 联动关系：无关 → 简单依赖 → 双向联动

## 四、技术方案（Unity实现）

### 4.1 项目结构

- **渲染方案：** Unity URP + 2D Renderer（支持光照，效果优于 Built-in）
- **地图系统：** Unity Tilemap（用于基础地板/墙壁）+ 自定义房间系统（管理 SwitchSlot）
- **物理方案：** Unity 2D Physics（2D Collider，不使用 Rigidbody 动态物理，用 TileMap Collider 静态碰撞）
- **动画方案：** Unity Animator + DOTween（用于切换动画和UI反馈）

### 4.2 房间数据结构

```csharp
// 预制件选项
[System.Serializable]
public class PrefabOption {
    public string id;           // 唯一标识
    public TileBase tile;        // 对应的Tilemap Tile
    public bool isWalkable;      // 是否可通行
    public bool isTransparent;   // 是否半透明（视觉参考）
    public Sprite icon;          // 槽位UI显示图标
}

// 槽位定义
[System.Serializable]
public class SwitchSlot {
    public Vector2Int gridPos;              // 在房间网格中的坐标
    public List<PrefabOption> options; // 可选预制件列表
    public int currentIndex;                 // 当前选中的选项索引
    public SlotType slotType;               // Toggle / Cycle / Conditional
    public string dependsOnSlotId;           // 依赖的槽位ID（Conditional时使用）
    public List<int> unlockedByState;        // 依赖槽位的哪些状态解锁本槽位
}

// 房间定义
[System.Serializable]
public class Room {
    public string id;                        // 房间唯一ID
    public string displayName;               // 显示名称
    public int width;                        // 房间网格宽度（格）
    public int height;                       // 房间网格高度（格）
    public List<SwitchSlot> switchSlots;     // 槽位列表
    public Vector2Int playerSpawnPos;        // 玩家出生点（格）
    public Vector2Int exitPos;                // 出口位置（格）
    public string nextRoomId;                // 下一房间ID
    public List<string> requiredRoomIds;      // 进入本房间的前置房间ID列表
    public ChapterType chapter;              // 所属章节
}

// 章节定义
public enum ChapterType { Ch1_FirstLight, Ch2_DeepDig, Ch3_LostPath }

// 关卡管理器
public class LevelManager : MonoBehaviour {
    public TextAsset levelDataJson;           // JSON格式关卡数据
    public Dictionary<string, Room> roomDict;
    // 提供房间加载、状态保存、重置功能
}
```

### 4.3 关卡编辑器建议

**推荐制作自定义关卡编辑器（EditorWindow）：**
- 在Unity Editor中嵌入房间编辑器
- 可视化编辑房间网格、放置槽位、配置预制件
- 一键导出为 JSON 或 ScriptableObject

**编辑器输出格式选择：JSON**
- 优势：人类可读、易于外部工具生成、可被其他平台解析
- 配合 Resources.Load<TextAsset> 加载

## 五、美术风格

### 5.1 推荐视觉风格

**参考方向：** Dead Cells × Gorogoa × 废弃设施氛围感

| 要素 | 建议 |
|------|------|
| **色调** | 低饱和度冷色调为主（深蓝灰、石墨色），槽位发光用青色/橙色点缀 |
| **整体氛围** | 废弃研究设施感，带有一点神秘感 |
| **UI风格** | 极简主义，半透明悬浮面板，无边框设计 |
| **槽位提示** | 槽位位置有微弱脉冲发光（青色），切换时发光增强 |
| **出口提示** | 连通时出口发出暖色脉冲光（橙黄） |

**调色板建议：**

| 用途 | 色值 |
|------|------|
| 背景/深色墙 | #1A1A2E |
| 地板 | #2D2D44 |
| 实墙 | #3D3D5C |
| 槽位发光 | #00D4FF（青色） |
| 出口发光 | #FF9500（橙色） |
| 文字/UI | #E0E0E0 |

### 5.2 音效/音乐

**最低成本方案：**
- **免费音效库：** Kenney.nl（免费游戏素材）、freesound.org（CC0音效）
- **AI生成音乐：** Suno / Udio（生成氛围感BGM，风格tag：ambient, mysterious, calm）
- **工具链：** Audacity（音频剪辑）+ Audiosprite（音效合并）

**音效建议：**
- 切换音效：短促的电子音（类似机器运转）
- 地板铺设成功：低沉的"咔哒"
- 出口连通：上升的提示音
- 背景：环境白噪音+低频ambient

## 七、风险点和应对

### 技术风险

| 风险 | 应对方案 |
|------|--------|
| Tilemap与SwitchSlot数据同步复杂 | 用ScriptableObject统一管理，所有切换操作走统一接口 |
| 房间序列化/反序列化出错 | 写单元测试覆盖所有房间数据加载路径 |
| 性能问题（大量房间+动画） | 使用对象池预加载相邻房间，动画用DOTween而非Animator |

### 设计风险

| 风险 | 应对方案 |
|------|--------|
| 槽位组合太多导致谜题失去挑战性 | 每个房间设计时明确"核心推理点"，避免枚举穷举式解法 |
| 玩家不知道槽位在哪里 | 槽位始终有视觉标记（脉冲光），且第一个房间强制教学 |
| 难度曲线失控（太难或太简单） | 第1章5个房间严格控制，每关由浅入深；邀请1~2名测试玩家 |

### 资源风险

| 风险 | 应对方案 |
|------|--------|
| 美术资源跟不上 | 全程使用placeholder素材，最终发布前统一替换 |
| 个人开发者精力不足 | 砍掉所有非必要功能（成就、收集、天气系统），聚焦核心切换机制 |

**按键映射：**

- R键：重置当前房间所有槽位
- ESC键：暂停菜单

## ⑤ 数值设计

### 5.1 房间难度权重计算

每个房间的难度通过以下公式量化：

```
房间难度 = (槽位数量 × 选项系数) + 联动复杂度 + 空间推理难度
```


| 参数 | 计算方式 | 说明 |
|------|----------|------|
| **槽位数量** | 1-4个 | 基础权重 |
| **选项系数** | 2选项=1.0 / 3选项=1.5 / 4选项=2.0 | 每增加一个选项，复杂度指数增长 |
| **联动复杂度** | 无关=0 / 单向依赖=2 / 双向联动=4 | 需要同时满足多槽位状态 |
| **空间推理** | 线性=0 / 非线性=2 / 多路径=4 | 需要考虑玩家移动顺序 |


### 5.2 各章节难度分布

| 章节 | 房间数 | 平均难度 | 难度范围 | 教学目标 |
|------|--------|----------|----------|----------|
| 第一章 | 5 | 2-4 | 2-5 | 基础切换概念 |
| 第二章 | 6 | 5-8 | 4-10 | 条件依赖 |
| 第三章 | 8 | 9-14 | 7-16 | 联动+伪装 |

### 5.3 槽位切换次数预期

基于难度合理控制玩家尝试次数：

| 难度等级 | 预期切换次数 | 最大允许(不禁用) | 激励重置点 |
|----------|-------------|-----------------|--------------|
| 简单 | 1-3次 | 无限制 | 即时满足感 |
| 中等 | 3-8次 | 无限制 | 思考后顿悟 |
| 困难 | 8-20次 | 无限制 | 多次试错学习 |

**设计原则：** 虽然允许无限次切换，但通过以下方式保持挑战感：
1. 每个房间有明确的"核心解法"（不止一种但有限）
2. 错误配置会导致明显的死路（视觉反馈）
3. 重置成本极低，鼓励试错

### 5.4 章节通关率目标

基于类似游戏数据设计的预期：

| 章节 | 目标通关率 | 预期卡点 | 辅助设计 |
|------|-------------|----------|----------|
| 第一章 | >95% | 无 | 强制教学 |
| 第二章 | 80-90% | Ch2-3~Ch2-4 | 提供hint（在房间显眼位置放提示碎片） |
| 第三章 | 70-85% | Ch3-4 / Ch3-6 | 可选的渐进式hint系统 |

## ⑥ 玩家体验曲线

### 6.1 前30分钟体验节奏

```
时间轴  │ 内容                                           │ 情绪
───────┼────────────────────────────────────────────────┼──────────
0-3min │ [1-1] 教学：单槽位二选一，理解槽位概念          │ 好奇
3-6min │ [1-2] 双槽位：需要同时切换两个槽位               │ 思考
6-10min│ [1-3] 三槽位：理解出口位置与配置的关系          │ 挑战
10-15min│ [1-4] 首次失败教学：按R重置，回顾观察          │ 顿悟
15-20min│ [1-5] 第一章结尾：引入CycleSlot                 │ 成就感
20-30min│ [2-1~2-2] 第二章开始：ConditionalSlot          │ 新鲜感
```

### 6.2 "顿悟时刻"设计

顿悟时刻指玩家突然理解某个机制的关键瞬间：

| 章节 | 顿悟时刻 | 设计方式 |
|------|----------|----------|
| Ch1 | "原来只是换个方向" | 第一次切换后发现墙变路，路径就在身后 |
| Ch2 | "原来要先开这个" | 门被另一槽位挡住，需要先切换那一个 |
| Ch3 | "表面看是这样" | 伪装地板/真假墙，需要透过表象看本质 |

### 6.3 机制引入节奏

```
第一章 │ 引入:ToggleSlot(双选)→双槽位联动→三槽位组合
───────┼────────────────────────��────────────────────
第二章 │ 引入:ConditionalSlot(条件解锁)→时序依赖→门控
───────┼─────────────────────────────────────────────
第三章 │ 引入:联动开关→伪装地板→空间欺骗→综合挑战
```

## ⑦ 失败/重试设计

### 7.1 无死亡无惩罚下的挑战感来源

**核心挑战来源：**

| 来源 | 实现方式 | 效果 |
|------|----------|------|
| **认知挑战** | 槽位配置的逻辑推理 | 成就感来自"想通了" |
| **空间直觉** | 想象切换后的房间结构 | 顿悟快感 |
| **最优解追求** | 设计多解法，鼓励最短步数 | 重玩动力 |
| **收集欲** | 隐藏成就/快速通关分数 | 完美主义驱动 |

### 7.2 防止"无脑乱按"的设计

虽然允许无限次切换，但通过以下设计试错成本：

| 设计 | 作用 |
|------|------|
| **观察时间成本** | 玩家需要花时间理解当前配置，然后再试 |
| **切换动画** | 0.2s动画让玩家看到每次变化，无法快速跳过 |
| **无快捷解法** | 组合空间大，穷举不现实（3槽位×3选项=27种组合） |
| **配置记忆** | 重置后忘记之前的配置，需要重新推理 |

### 7.3 重置激励机制

| 机制 | 说明 |
|------|------|
| **R键即时响应** | 无加载，房间瞬间恢复初始状态 |
| **提示系统（可选）** | 卡关太久提供渐进式hint | 
| **重置计数器** | 记录本房间重置次数，完成后显示 | 
| **"再来一次"成就感** | 鼓励玩家把这当成学习过程而不是失败 |

**HUD位置：**

- 右上角：设置按钮 + 帮助按钮
- 右下角：小键盘提示（E切换/Q反向/R重置）

### 8.2 槽位交互UI

**槽位提示（玩家靠近时显示）：**

```
┌─────────────────────────────────┐
│  ◉ 槽位 1/3          [E]切换   │
│  当前: 地板        [Q]反向    │
│  ━━━━━━━━━━                   │
│  ① 地板 ② 墙 ③ 门          │
│  ● ○ ○                         │
└─────────────────────────────────┘
```

**设计原则：**
- 玩家距离槽位2格内时显示提示
- 半透明黑色底+白色文字
- ��前选中项高亮显示
- 切换后提示淡入淡出

### 8.3 界面视觉规范

## ⑨ 音效与音乐需求清单

### 9.1 音效清单

| 事件 | 描述 | 时长 | 参考风格 |
|------|------|------|----------|
| **切换成功** | 机械卡扣声 | 0.2s | 电子音+机械感 |
| **多个槽位切换** | 连续咔哒声(0.1s间隔) | 0.4s | 机关启动 |
| **重置** | 卷轴倒带/数据清除声 | 0.3s | 科技感 |
| **出口连通** | 通电/上升音效 | 0.5s | 成就感提示音 |
| **踏入无效区域** | 轻微阻碍音/撞击音 | 0.15s | 轻柔的拒绝音 |
| **进入新房间** | 开门/空间转换 | 0.6s | 场景转换 |
| **章节完成** | 完成音效 | 1.0s | 成就结算 |
| **获得成就** | 解锁音 | 0.8s | 正向激励 |


### 9.2 环境音效

| 场景 | 描述 | 循环方式 |
|------|------|----------|
| **室内背景** | 低沉的设施环境音（嗡嗡声、远处水滴） | 循环 |
| **走廊回声** | 脚步回声（如果有） | 触发 |
| **电力蜂鸣** | 槽位附近的电流声 | 循环（靠近时渐强） |

### 9.3 音乐需求

| 场景 | 风格 | 参考 | 长度 |
|------|------|------|------|

## ⑩ 运营/发布计划

### 10.1 平台选择

| 平台 | 优势 | 劣势 | 建议 |
|------|------|------|------|
| **Itch.io** | 完全免费、易上手、 Indie友好 | 流量较低 | 第一优先：免费发布 |
| **Steam** | 大流量、变现能力强 | 审核费$100、需商店页 | 第二阶段：付费发布 |
| **移动端(iOS/Android)** | 日活高、触达易 | 适配成本、买量贵 | 暂不考虑（首版） |
| **Nintendo Switch** | 独立游戏聚集地 | 审核严格、移植成本 | 后续调研 |

### 10.2 发布策略

**Itch.io阶段（推荐）：**
- 发布时间：开发完成即发布
- 价格：免费（或Pay What You Want，最低$1）
- 目的：收集反馈、建立玩家社区

**Steam阶段（进阶）：**
- 定价策略：$4.99（初版）→ $6.99（正式版）
- 首月促销：EA期间15% OFF
- 理想发布窗口：大型促销活动前后（如Summer Sale）

### 10.3 宣传节点

| 时间点 | 动作 | 内容 |
|--------|------|------|
| T-3个月 | 建立社媒 | Twitter/X账号、Discord服务器 |
| T-2个月 | 预告片 | 30秒预告片+Steam页面 | 
| T-1个月 | 试玩版 | 发布Demo、邀请主播试玩 |
| T-1周 | 发力宣传 | KOL推广、新闻稿投递 |
| 发布日 | 正式发布 | 多平台发布公告 |
| T+2周 | 更新1 | 首日补丁+Bug修复 |
| T+1月 | 收集反馈 | 基于玩家反馈优化 |

### 10.4 关键指标

| 指标 | 目标(Itch.io) | 目标(Steam) |
|------|---------------|-------------|
| 下载量 | 1,000+ (首月) | 500+ (首月) |
