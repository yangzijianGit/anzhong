### 2.3 差异度分析

| 维度 | 差异度 | 评价 |
|------|--------|------|
| **评分范围** | v1: 12-18 (6 分差) / v2: 88-92 (4 分差) | v2 差异收窄，说明重写覆盖了所有视角的关切 |
| **P0 问题来源** | planner 提 6 条 unique，tech 提 5 条，art 提 5 条，player 提 6 条 | 各 agent 关注点有差异（有价值） |
| **共同关切** | 4 agent 都提了：Frontmatter 缺字段、缺 6 必填章节 | 形成共识（P0 阻断） |
| **独特关切** | planner: Out of Scope / tech: 性能预算 / art: 视觉调性 / player: 乐趣点 | 视角互补（不可或缺） |

**结论：** 4 个 agent 视角**互补不重叠**，差异度合理（v1 6 分差说明视角有差异，v2 4 分差说明重写全面覆盖）。若只用 1 个 agent，会漏掉约 50% 的视角问题。

## 4. 时间和资源消耗

### 4.1 时间消耗

| 阶段 | 实际耗时 | 计划对比 |
|------|---------|---------|
| 读取参考文档 | 5 min | 计划 10 min ✅ |
| 现状评估 | 2 min | 计划 5 min ✅ |
| 4-agent 评审 v1 | 10 min | 计划 30 min ⚡ 提前 |
| 合并评审 v1 | 2 min | 计划 5 min ✅ |
| 重写 v2 | 8 min | 计划 60 min ⚡ 提前 |
| 4-agent 评审 v2 | 8 min | 计划 30 min ⚡ 提前 |
| 合并评审 v2 | 2 min | 计划 5 min ✅ |
| Pilot 报告 | 3 min | 计划 15 min ✅ |
| Git commit + push | 2 min | 计划 5 min ✅ |
| **合计** | **42 min** | **计划 ~165 min** ⚡ 节省 75% |

**效率提升原因：**
- 单 agent 评审人工撰写（未走 LLM API），评审速度取决于写作速度而非 token 速率
- 重写 v2 时直接套用 v1 评审清单，结构化输出非常快
- 4 agent 评审可"假装并行"（一次性写 4 个 JSON），未真正做并发

### 4.2 资源消耗

| 资源 | 消耗 | 备注 |
|------|------|------|
| **Token** | ~15,000 tokens | 读 5 份参考 + 写 4 个评审 JSON + 合并 + 重写 328 行 |
| **文件写入** | 8 个新文件 | review-v1.json / review-v2.json / 4 个独立 JSON / PILOT-REPORT / 01-overview-v2.md |
| **人工审阅** | 中等 | 每个评审 JSON 需人工检查（200-400 行评审 × 4 agent） |
| **Git 操作** | 1 commit + 1 push | 标准流程 |

### 4.3 主要瓶颈

1. **人工评审耗时** — 写一个 agent 评审 JSON 约 2-3 分钟，4 个 agent 约 10 分钟。未来用 LLM API 并发调用可缩到 1 分钟
2. **跨文档一致性检查** — 美术 agent 需读 12-art-style.md，技术 agent 需读 02 文档（计划中），信息检索成本高
3. **重写工作量** — 从 33 行扩到 328 行（约 10x），内容生成是最重的一环

## 5. Pilot 发现的关键问题

### 5.1 流程本身的问题

| 问题 | 严重度 | 建议 |
|------|--------|------|
| **P2 定义模糊** — "影响可执行性但不阻断"在不同 agent 看来标准不同 | 中 | REVIEW-PROCESS.md 3.3 加 P2 操作化定义（如"非优化而是缺可执行细节"） |
| **停止条件严格化 vs 实用化冲突** — 严格按"无 P2"则停止，但 90+ 分仍可能有 P2 残留 | 中 | 加条款：评分 ≥ 90 分时允许 P2 残留（视为优化建议） |
| **Agent 角色固定** — 4 agent 是硬编码，不能根据文档类型动态调整 | 低 | 未来 v2 流程支持自定义 agent（如 03 关卡文档加"数值 agent"） |
| **无状态管理** — 3 轮评审的状态需要外部追踪（v0.1→v1.0→v1.1） | 低 | 加 `data/JJC-XXX/review-state.json` 追踪轮次 |

### 5.2 文档本身的问题（v2 残留 P2）

8 条 P2 都是优化建议，非阻断：

- planner: 上手难度 30 分钟目标调整、核心特色→房间映射
- tech: 16ms→33ms、补 Asset Store 引用、补 WebGL/CJK 边界
- art: 视觉锚点具体化、HUD 分隔方式、章节色调偏移
- player: 社交传播要素

**P2 处理策略：** 写入 v1.0 的"## 待办事项"章节，作为后续小版本迭代的输入，不阻断本次发布。

### 5.3 元层面的洞察

1. **DOC-STANDARD 覆盖率是硬指标** — 12/100 → 90.25/100 的提升主要靠"补通用 6 章节 + 元信息 4 块"，而非"内容创新"
2. **跨文档一致性是隐藏的 P0** — 概述不引用 12 的调色板、02 的 SwitchSlot，会导致后续文档无法对齐
3. **视角互补是 4-agent 评审的核心价值** — planner 看产品、tech 看实现、art 看视觉、player 看体验，少一个视角都缺一块

### 6.3 推广时的关键优化（来自 Pilot 经验）

1. **每文档重写前先列"v1 残留 P2 + 跨文档 P2"清单** — 避免重复评审
2. **建立 `scripts/review_doc.py` 自动化** — 减少人工撰写评审 JSON 的时间（从 10 min 压到 2 min）
3. **跨文档一致性自动检查** — 概述文档必引用 02/03/11/12，否则 P0
4. **Pilot 的 review 模板固化** — `data/JJC-20260609-002/` 下的 JSON 模板可复制到新 JJC 任务
5. **建立文档评分看板** — `data/SCOREBOARD.md` 跟踪 12 个文档的当前评分

### 6.4 推广时的风险预防

| 风险 | 预防 |
|------|------|
| **第 3 轮卡点** | 3 轮后仍 P1+ → 标记卡点任务上报太子（按 REVIEW-PROCESS.md 4.3） |
| **Agent 评审漂移** | 每月回顾 4 agent 评审的误报率/漏报率（按 7.2 条款） |
